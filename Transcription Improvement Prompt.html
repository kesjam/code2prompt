Guidelines for Creating a Whisper Prompt:

  1. Provide Contextual Sentences:

    • Start with a sentence or two that sets the scene for the audio content.
    • Example: For a business meeting, you might use, "In today's meeting, we'll discuss the quarterly financial results and upcoming projects."

  2. Include Relevant Vocabulary:

    • If the audio contains technical terms, names, or jargon, include them in the prompt.
    • Example: For a medical lecture, "The discussion covers cardiomyopathy, hypertension, and echocardiogram results."

  3. Keep It Concise:

    • The prompt should be brief—typically one to three sentences.
    • Long prompts may not significantly improve transcription quality and can slow down processing.

  4. Match the Audio Style and Language:

    • Use the same language and tone as the audio.
    • If the audio is formal, keep the prompt formal; if it's casual, a conversational prompt is appropriate.

  5. Avoid Overloading with Instructions:

    • Unlike prompts for language models like GPT-4, the Whisper prompt should not contain instructions or guidelines.
    • Do not include directives like "remove filler words" or "correct grammar."

Example Prompts:

  • Casual Conversation:
    • "Hey, did you finish watching the latest episode of the series? I can't believe what happened!"
  • Technical Discussion:
    • "Today we'll explore the implementation of neural networks and backpropagation algorithms in deep learning."
  • Lecture or Presentation:
    • "Welcome to the seminar on sustainable energy solutions and their impact on global economics."

  • Improves Accuracy:
    • Helps the model anticipate and correctly transcribe specialized terms or names.
  • Disambiguates Audio:
    • Provides context that can clarify homophones or unclear speech segments.
  • Sets the Topic:
    • Gives the model an understanding of the subject matter, improving overall transcription quality.

1. Overview of SQLite

SQLite is a lightweight, serverless, file-based relational database management system (RDBMS). It's embedded directly into the application, meaning all the data is stored locally on the user's device.

Key Characteristics:

  • Local Storage: Data is stored in a single file on the local filesystem.
  • Zero Configuration: No setup or administration is required; it works out of the box.
  • Lightweight and Fast: Ideal for applications with moderate amounts of data and single-user access.
  • No Network Dependency: Functions without an internet connection.

2. Overview of Supabase

Supabase is an open-source Firebase alternative that provides a hosted PostgreSQL database along with additional backend services like real-time data synchronization, authentication, storage, and serverless functions.

Key Characteristics:

  • Cloud-Based Storage: Data is stored on remote servers accessible over the internet.
  • Full PostgreSQL Database: Offers advanced SQL features, data types, and extensibility.
  • Additional Services:
    • Real-Time Capabilities: Live data synchronization across clients.
    • Authentication: Built-in user management and security features.
    • Storage: For files like images or audio.
    • Edge Functions: Serverless functions for backend logic.
  • Scalability: Can handle large amounts of data and concurrent users.
  • Requires Internet Connection: Dependent on network availability.

3. Detailed Comparison

A. Data Storage Location

  • SQLite: Data is stored locally on the user's device.
  • Supabase: Data is stored in the cloud on Supabase's servers.

Implications for Your Project:

  • SQLite:

    • Pros:
      • Quick read/write operations due to local storage.
      • Works offline without internet connectivity.
    • Cons:
      • Data is only accessible from the device it was created on.
      • Not suitable for multi-device synchronization.

  • Supabase:

    • Pros:
      • Data can be accessed from multiple devices.
      • Facilitates data backup and recovery.
    • Cons:
      • Requires a consistent internet connection.
      • Potential latency due to network communication.

B. Accessibility and Synchronization

  • SQLite: Suited for single-user applications with no need for data sharing or synchronization.
  • Supabase: Designed for applications requiring real-time data synchronization and multi-user access.

Implications:

  • SQLite:

    • Ideal if the application is intended for use by a single user on one device.
    • No built-in mechanisms for data synchronization across devices.

  • Supabase:

    • Enables features like accessing dictations from multiple devices.
    • Supports collaborative features if you plan to expand the app for multiple users.

C. Scalability

  • SQLite:

    • Best for applications with moderate data size and low concurrency.
    • Performance may degrade with very large datasets or high concurrent access.

  • Supabase:

    • Built to handle larger datasets and higher loads.
    • Scales automatically to accommodate more data and users.

Implications:

  • SQLite:

    • Sufficient if you expect the data size to remain relatively small.
    • Limited scalability may become a bottleneck as the app grows.

  • Supabase:

    • Suitable if you plan to scale the application to support many users or large amounts of data.

D. Features and Functionality

  • SQLite:

    • Provides basic database functionalities.
    • Does not include advanced features like real-time subscriptions, authentication, or file storage.

  • Supabase:

    • Offers a suite of backend services:
      • Authentication: User sign-up, login, and security management.
      • Real-Time Updates: Live data synchronization.
      • Storage Services: Upload and manage files.
      • Serverless Functions: Run backend code without managing servers.

Implications:

  • SQLite:

    • You'll need to implement additional features (like authentication) separately if required.
    • Simpler setup if you don't need advanced backend services.

  • Supabase:

    • Provides a comprehensive backend solution.
    • Simplifies the addition of advanced features without building them from scratch.

E. Complexity and Setup

  • SQLite:

    • Very easy to set up; no server configuration required.
    • Embedded directly into your application code.

  • Supabase:

    • Requires setting up an account and configuring your project on the Supabase platform.
    • Involves managing API keys and understanding their SDKs or RESTful APIs.

Implications:

  • SQLite:

    • Faster initial development due to simplicity.
    • Less overhead for small or simple applications.

  • Supabase:

    • Steeper learning curve for initial setup.
    • Additional steps required for configuration and security management.

F. Offline Access

  • SQLite:

    • Fully functional without internet connectivity.
    • Ideal for users who may not always have internet access.

  • Supabase:

    • Dependent on an internet connection to interact with the database.
    • Offline functionality requires additional implementation (e.g., local caching).

Implications:

  • SQLite:

    • Users can create and access dictations anytime, regardless of connectivity.
    • Data remains on the local device.

  • Supabase:

    • Users must be online to save or retrieve dictations.
    • May need to implement offline caching strategies for better user experience.

G. Security and Authentication

  • SQLite:

    • Limited to file system permissions on the local device.
    • No built-in user authentication or access control beyond what's provided by the operating system.

  • Supabase:

    • Provides robust security features:
      • User Authentication: Securely manage user accounts.
      • Row-Level Security Policies: Fine-grained access control over data.
      • Encrypted Connections: Data transmitted over secure channels (HTTPS).

Implications:

  • SQLite:

    • Adequate for applications where data privacy is managed locally.
    • Less suitable if you require user authentication or need to protect sensitive data.

  • Supabase:

    • Better suited for applications requiring secure user authentication.
    • Essential if handling sensitive data that must comply with security standards.

3. Implementing SQLite in Your Application

A. Setting Up SQLite

Python includes the sqlite3 module by default, so you don't need to install anything extra.

Creating the Database and Table:

In data_manager.py:

python
import sqlite3

def create_connection():
    conn = sqlite3.connect('data/dictations.db')
    return conn

def create_table():
    conn = create_connection()
    cursor = conn.cursor()
    cursor.execute('''
        CREATE TABLE IF NOT EXISTS dictations (
            id INTEGER PRIMARY KEY AUTOINCREMENT,
            date_time TEXT,
            subject TEXT,
            content TEXT,
            audio_file_path TEXT,
            dictation_type TEXT
        )
    ''')
    conn.commit()
    conn.close()

Call create_table() when your application starts to ensure the table exists.

def transcribe_audio(audio_file_path):
    audio_file = open(audio_file_path, 'rb')
    transcript = openai.Audio.transcribe("whisper-1", audio_file)
    return transcript['text']

    prompt = f"""

def clean_transcription(transcription, dictation_type):
    prompt = generate_prompt(transcription, dictation_type)
    response = openai.Completion.create(
        engine="gpt-4",
        prompt=prompt,
        max_tokens=1024,
        n=1,
        stop=None,
        temperature=0.5,
    )
    cleaned_text = response['choices'][0]['text']
    return cleaned_text.strip()
